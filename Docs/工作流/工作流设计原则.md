# AI 工作流助手（类 Cursor / Claude Code）搭建指导文档

> 本文档面向**工具层 / Agent 层**的实现者，目标是构建一个“会思考、会检索、会调用工具、会迭代执行”的 AI 工作流助手。  
> 不涉及 LLM 内部实现，仅关注 **API 调用、工具编排、状态管理与工程落地**。

---

## 一、核心认知前提（必须先对齐）

### 1. 这是一个「多次 LLM 调用」系统
- 每一次“思考 / 决策” = 一次 LLM API 调用
- 每一次“工具执行” = 一次外部动作
- 整个任务 = **LLM 调用 × N + 工具调用 × M**

不存在：
- 一次 LLM 调用就完成“检索 → 读文件 → 改代码 → 跑测试”

### 2. Tool Use ≠ 一次调用反复执行
Tool Use 的本质是：
- LLM 输出结构化 action
- 你执行 action
- **再把结果喂回模型，重新调用 LLM**

## 二、整体系统架构（推荐）

┌─────────────┐
│   Frontend  │  UI / CLI / IDE 插件
└─────┬───────┘
│ 用户请求
┌─────▼───────┐
│ Orchestrator│  状态机 / Agent Loop
└─────┬───────┘
│ 多次调用
┌─────▼───────┐
│     LLM     │  负责“下一步该做什么”
└─────┬───────┘
│ tool_call
┌─────▼───────┐
│   Tools     │  文件 / 搜索 / 执行命令
└─────┬───────┘
│ 结果
┌─────▼───────┐
│   Memory    │  上下文压缩 / 摘要 / 索引
└─────────────┘

---

## 三、最小可行工作流（MVP）

### 标准执行链路

用户请求
→ 解析意图
→ 生成计划
→ 检索上下文
→ 阅读关键内容
→ 生成修改方案
→ 应用修改
→ 验证结果
→ 总结 / 继续迭代

---

## 四、模块拆分与实现要点

### 1. Orchestrator（核心控制器）

职责：
- 维护任务状态
- 决定下一步调用 LLM 还是工具
- 控制循环终止条件

推荐形式：
- 显式状态机（推荐：LangGraph 思路）
- 或 while-loop + step 类型判断

**状态示例（State Schema）**
```json
{
  "goal": "用户目标",
  "constraints": [],
  "plan": [],
  "current_step": "retrieve",
  "read_files": [],
  "file_summaries": {},
  "pending_actions": [],
  "last_tool_result": null,
  "done": false
}


⸻

2. LLM 节点设计（不是一个“万能 Prompt”）

每次 LLM 调用都应该目的单一。

常见 LLM 节点类型
	1.	Intake / Intent Parsing

	•	输入：用户原始请求
	•	输出：
	•	明确目标
	•	不做的事
	•	成功标准

	2.	Planner

	•	输出：可执行步骤列表
	•	可展示给用户（≠ chain-of-thought）

	3.	Decision / Next Action

	•	输出：

{
  "action": "search_files",
  "args": { "query": "auth middleware" }
}

	4.	Patch Proposal

	•	输出：unified diff / patch schema

	5.	Result Summarizer

	•	输出：本轮做了什么、状态是否完成

⸻

3. Tools 设计（必须可控、可审计）

必备工具集合（类 Cursor）
	•	文件系统
	•	list_files
	•	search_files（rg / index）
	•	read_file(path, start, end)
	•	代码修改
	•	apply_patch(diff)
	•	执行验证
	•	run_command(cmd)
	•	辅助
	•	git_status
	•	git_diff

原则
	•	所有工具必须：
	•	显式输入
	•	显式输出
	•	可回放 / 可日志化

⸻

4. 上下文与 Memory 策略（非常关键）

不要做的事
	•	❌ 每次把整个 repo 塞给模型
	•	❌ 每轮无限增长上下文

推荐三层记忆
	1.	短期上下文

	•	当前步骤相关文件片段
	•	最近一次命令输出

	2.	中期摘要

	•	每个已读文件 → 结构化摘要
	•	模块职责、关键函数

	3.	长期索引

	•	文件路径 / 语言 / embedding
	•	启动时构建，查询时使用

⸻

五、代码修改的正确姿势

不要
	•	让模型输出“完整文件内容”
	•	直接覆盖文件

要做
	•	使用 diff / patch
	•	应用前校验
	•	应用后回读关键片段

Patch 示例

--- a/auth/middleware.py
+++ b/auth/middleware.py
@@ -42,6 +42,10 @@ def auth_required(req):
     if not token:
         return unauthorized()

+    if is_expired(token):
+        refresh(token)
+
     return next(req)


⸻

六、验证与迭代闭环

没有验证，就没有“可靠感”

推荐策略：
	•	改完必跑：
	•	lint / typecheck
	•	unit tests（如有）
	•	失败输出 → 再次喂给 LLM → 修复

这是 Cursor / Claude Code 最核心的“像人”的地方。

⸻

七、UI / 交互层设计建议

分三条信息流
	1.	Plan

	•	当前目标
	•	执行步骤

	2.	Execution Logs

	•	调用了哪些工具
	•	读了哪些文件
	•	跑了哪些命令

	3.	Changes

	•	diff 视图
	•	文件列表

不展示
	•	原始 chain-of-thought
	•	内部 prompt

⸻

八、一次完整任务的调用次数示例

以“修一个 Bug”为例：
	1.	LLM：解析需求
	2.	LLM：生成计划
	3.	Tool：搜索文件
	4.	LLM：决定读哪些文件
	5.	Tool：读文件
	6.	LLM：生成 patch
	7.	Tool：apply_patch
	8.	Tool：run tests
	9.	LLM：根据失败修复（可选）
	10.	LLM：总结

👉 8–15 次 LLM 调用是正常的

⸻

九、实现顺序建议（落地路线）

Phase 1：CLI Agent（最快验证）
	•	单仓库
	•	少量工具
	•	明确 loop

Phase 2：Memory + 索引
	•	文件摘要缓存
	•	embedding + rg 混合检索

Phase 3：IDE / UI
	•	VS Code / Web UI
	•	Diff 可视化
	•	步骤展示

Phase 4：安全与策略
	•	命令白名单
	•	文件写入限制
	•	自动回滚
